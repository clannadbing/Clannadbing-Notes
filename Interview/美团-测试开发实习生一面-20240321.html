1. 测试和开发难点

测试难点：
1.覆盖率问题：测试所有可能情况和路径可能是不可行的，因此选择适当的测试用例以获得最大的覆盖率是挑战之一。
2.自动化测试的复杂性：自动化测试是必要的，但编写和维护自动化测试脚本可能很复杂。特别是对于一些交互复杂或动态性强的应用程序，编写稳健的自动化测试脚本更为困难。
3.环境管理：测试环境与生产环境之间的不一致可能导致一些问题被忽略或无法重现。
4.不确定性：软件系统的不确定性使得测试变得更加困难。例如，多线程、异步操作等都增加了测试的不确定性。
5.性能和负载测试：对于高负载或大规模系统，进行有效的性能和负载测试是非常具有挑战性的。
xxxxxxxxxx
开发难点：
1.复杂性管理：软件系统的复杂性可能导致代码难以理解和维护。
2.技术选型：选择适当的技术栈、框架和工具对于项目的成功至关重要，但是技术选型可能会受到多种因素的影响，包括团队技能、项目需求和预算限制等。
3.设计决策：设计良好的软件架构是系统的基础，但设计决策可能会受到需求变更、时间压力和团队约束等因素的影响。
4.代码质量：保持良好的代码质量对于项目的长期可维护性至关重要，但在日常开发中往往面临时间压力和需求变更等挑战。
5.集成和部署：整合不同的模块或组件，并确保系统能够顺利部署和运行是开发过程中的一个重要环节，但它可能会受到环境不一致、依赖管理和配置管理等因素的影响。

11. 接口信息

xxxxxxxxxx
接口信息通常包括接口的URL、请求方法、参数、响应内容等。以下是一些常见的接口信息及其说明：


1. URL（Uniform Resource Locator）：接口的URL指定了接口的位置。它表示了访问接口所需的资源路径和参数。例如：`https://api.example.com/user/login`
2. 请求方法（Request Method）：请求方法指定了客户端对服务器资源的请求类型。常见的请求方法包括GET、POST、PUT、DELETE等。例如：POST
3. 参数（Parameters）：参数是向接口发送的数据，通常包括查询参数、请求体参数等。查询参数出现在URL中，而请求体参数包含在请求的主体中。例如：用户名、密码等。
4. 请求头（Request Headers）：请求头包含了关于请求的元数据，如Content-Type、Authorization等。例如：Content-Type: application/json
5. 请求体（Request Body）：请求体包含了请求的主体数据，通常在POST、PUT等请求方法中使用。例如：`{"username": "example", "password": "password123"}`
6. 响应状态码（Response Status Code）：响应状态码表示了服务器对请求的处理结果。常见的状态码包括200（成功）、404（未找到）、500（服务器内部错误）等。例如：200
7. 响应头（Response Headers）：响应头包含了关于响应的元数据，如Content-Type、Cache-Control等。例如：Content-Type: application/json
8. 响应体（Response Body）：响应体包含了服务器返回的数据，通常是JSON、XML等格式。例如：`{"status": "success", "user_id": "12345"}`

12. GET和POST请求的区别？

xxxxxxxxxx
1. 作用不同 GET用于从服务端获取资源 POST一般用来向服务器端提交数据 
2. 参数传递方式不同 GET请求的参数一般写在URL中，且只接受ASCII字符 POST请求参数一般放在请求体中，对于数据类型也没有限制 
3. 安全性不同 因为参数传递方式的不同，所以两者安全性不同，GET请求的参数直接暴露在URL中，所以更不安全，不能用来传递敏感信息。 
4. 参数长度限制不同 GET传送的数据量较小，不能大于2KB。 POST传送的数据量较大，一般被默认为不受限制。 HTTP 协议没有 Body 和 URL 的长度限制，对 URL 限制的大多是浏览器和服务器的原因。 
5. 编码方式不同 GET 请求只能进行 URL 编码（application/x-www-form-urlencoded） POST 支持多种编码方式（application/x-www-form-urlencoded 或 multipart/form-data。为二进制数据使用多种编码。）
6. 缓存机制不同 GET 请求会被浏览器主动cache，而 POST 不会，除非手动设置。 GET 请求参数会被完整保留在浏览器历史记录里，而 POST 中的参数不会被保留。 GET 产生的 URL 地址可以被 保存为书签，而 POST 不可以。 GET 在浏览器回退时是无害的，而 POST 会再次提交请求。 
7. 时间消耗不同 GET 产生一个 TCP 数据包； POST 产生两个 TCP 数据包。 对于 GET 方式的请求，浏览器会把 header 和 data 一并发送出去，服务器响应 200（返回数据）；而对于 POST，浏览器先发送 Header，服务器响应 100 continue，浏览器再发送 data，服务器响应 200 ok（返回数据） 
8. 幂等 意思是多次执行相同的操作，结果都是「相同」的。 GET 方法就是安全且幂等的，因为它是「只读」操作，无论操作多少次，服务器上的数据都是安全的，且每次的结果都是相同的。 POST 因为是「新增或提交数据」的操作，会修改服务器上的资源，所以是不安全的，且多次提交数据就会创建多个资源，所以不是幂等的。

13. TCP与UDP的概念，特点，区别和对应的使用场景？

xxxxxxxxxx
TCP与UDP的概念 
    TCP（传输控制协议）是一种面向连接的、可靠的、基于字节流的传输层通信协议。 
    UDP（用户数据报协议）为应用程序提供了一种无需建立连接就可以发送封装的IP数据包的方法。 
    
特点 
    TCP：面向连接，传输可靠，传输形式为字节流，传输效率慢，所需资源多。 
    UDP：无连接、传输不可靠、传输形式为数据报文段，传输效率快，所需资源少。
    
对应的使用场景
TCP常用于要求通信数据可靠场景（如网页浏览、文件传输、邮件传输、远程登录、数据库操作等）。 
UDP常用于要求通信速度高场景（如域名转换、视频直播、实时游戏等）。

14. 客户端和服务端具体交互过程

xxxxxxxxxx
1. 建立连接：客户端向服务端发起连接请求。对于TCP协议，这通常是通过三次握手来完成的，即客户端发送SYN包（同步序列编号），服务端返回SYN-ACK包（同步-确认），最后客户端发送ACK包（确认）。这样就建立了双方之间的连接。
2. 发送请求：客户端向服务端发送请求。请求通常包含了请求的方法（GET、POST等）、路径、请求头、请求体等信息，这取决于具体的应用和协议。例如，在HTTP协议中，客户端通过发送HTTP请求来向服务端请求资源。
3. 处理请求：服务端收到请求后，会根据请求的内容进行相应的处理。这可能涉及到查询数据库、执行业务逻辑、调用其他服务等操作，最终产生需要返回给客户端的响应数据。
4. 发送响应：服务端处理完请求后，会将生成的响应数据发送给客户端。响应通常包含了响应状态码、响应头、响应体等信息，用于告知客户端请求的处理结果。例如，在HTTP协议中，服务端通过发送HTTP响应来返回请求的结果。
5. 接收响应：客户端接收到服务端发送的响应后，会进行相应的处理。这可能涉及到解析响应数据、处理响应状态码、处理响应头、处理响应体等操作。
6. 关闭连接：在完成请求和响应的交互后，客户端和服务端会根据具体的协议和业务逻辑来决定是否关闭连接。对于HTTP协议，默认情况下，每个请求-响应交互都会关闭连接，但也可以通过设置连接保持（Keep-Alive）来保持连接持久化，以减少连接建立的开销。

15. Cookie和Session是什么？有什么区别？

xxxxxxxxxx
Cookie 和 Session 都用于管理用户的状态和身份, Cookie 通过在客户端记录信息确定用户身份， Session 通过在服务器端记录信息确定用户身份


存储位置：Cookie 数据存储在用户的浏览器中，而 Session 数据存储在服务器上。 
数据容量：Cookie 存储容量较小，⼀般为几 KB。Session 存储容量较大，通常没有固定 限制，取决于服务器的配置和资源。
安全性：由于 Cookie 存储在用户浏览器中，因此可以被用户读取和篡改。相比之下， Session 数据存储在服务器上，更难被用户访问和修改。 
传输方式：Cookie 在每次 HTTP 请求中都会被自动发送到服务器，而Session ID 通常通 过 Cookie 或 URL 参数传递。
